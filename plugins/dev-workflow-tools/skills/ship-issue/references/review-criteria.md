# レビュー観点（ship-issue 共有）

このファイルは ship-issue のレビュー担当サブエージェントが従う観点を定義する。
PR の差分に対して、独立したレビュアーとして、実装者とは別の視点で指摘する。甘くしない。

## 見るべき観点

- **ガイドライン遵守**: プロジェクトの規約（`CLAUDE.md` / README / コントリビューションガイド等）に従っているか。特にテスト実行方法など明文化されたルールがあれば守られているか。
- **コード品質 / ベストプラクティス**: そのプロジェクトの言語・フレームワークの慣習や既存コードのパターンに沿っているか。周囲のコードと粒度・命名・スタイルが揃っているか。
- **バグ / セキュリティ**: null 参照、境界条件、例外処理漏れ、認証・権限チェック漏れ、機密情報の露出。
- **パフォーマンス**: 不要な再計算・再描画、N+1、重い同期処理、リソースリーク（リスナー/購読の解除漏れ、ハンドルやコネクションの未クローズ等）。
- **保守性 / 可読性**: 過度な複雑さ、責務の混在、テストの欠落・薄さ。
- **設計 / アーキテクチャ**: 層の分離、依存方向、拡張余地の妥当性。

## コメントの書き方

- **インラインコメントの構成**:
  - **結論を 1 行で先に**: 各コメント冒頭で指摘の要点を簡潔に。
  - **理由 → 具体的な修正案**: なぜそう判断したか、どう直すかを続けて書く。
  - **指摘中心**: 修正提案・バグの可能性・可読性の問題など、具体的で実行可能な改善点に絞る。
- **ポジティブなフィードバック**:
  - インラインで触れるのは特筆すべき良い設計・参考になる実装に限定。
  - 全体的に良かった点は、レビューの**まとめコメント（`pr-batch-review` JSON の `body`）**に集約する。
- **言語**: 日本語。
- **粒度**: 些末な好みの押し付けより、PR の目的に対して効くものを優先。指摘が無ければ無理に作らず「指摘なし」とする。

## 補足

- 最新のベストプラクティスが必要なら `WebSearch` / `WebFetch` で確認してよい。
- 行番号は**変更後（新しい版）の行番号**を使う（`pr-batch-review` の仕様。`side: RIGHT` が自動付与される）。変更行以外へのコメントはスキップされる点に注意。
