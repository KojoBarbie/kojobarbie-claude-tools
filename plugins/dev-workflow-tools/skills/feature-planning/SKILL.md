# Feature Planning

ふわっとした機能アイデアを、実装可能な GitHub Issue 群へ落とし込むためのスキル。
「深掘り → 親 Issue（ユーザーストーリー＋受け入れ条件）→ サブ Issue 分割 → GitHub へ作成・リンク」という流れを 1 本のセッションで完走する。

このスキルは**特定のリポジトリに依存しない**。リポジトリ名・ラベル・オーナーはすべて実行時に `gh` から取得するので、別の個人開発プロジェクトにディレクトリごとコピーすればそのまま動く。

## いつ使うか

ユーザーが「こういう機能が欲しい」「次にこれを作りたい」と、**まだ要件が曖昧な状態**で話し始めたとき。
逆に、仕様が完全に固まっていて単に文章を整形するだけのとき、あるいは単発の小さなバグ修正のときは、わざわざこのフローを回す必要はない。

## 前提チェック（最初に一度だけ）

作業を始める前に、サイレントに環境を確認する。ユーザーに逐一報告する必要はないが、欠けていたら止めて伝える。

1. `gh auth status` が通っているか（認証なしでは Issue を作れない）
2. `gh repo view --json owner,name` でリポジトリを特定できるか（git リポジトリ内にいるか）
3. リポジトリで sub-issue API が使えるか（GitHub の組織/個人リポジトリで利用可能。失敗時の代替は後述）

文脈理解のため、存在すれば以下を**軽く眺める**（なくても問題ない。あくまで質問を的確にするための材料）:
- `.kiro/steering/product.md` `tech.md` `structure.md`（プロダクト方針・技術スタック・構造）
- `README.md`、`CLAUDE.md`

これらは「このプロジェクトでは何が自然な実装か」を掴むための参照であって、このスキルは Kiro spec フローからは独立して動く。spec ファイルは生成しない。

## フェーズ 1: 深掘り（1 問ずつ対話的に）

目的は、**ユーザーストーリーと受け入れ条件が書ける程度に機能を具体化すること**。網羅的な仕様書を作ることではない。

### 進め方

- **1 回につき 1 問**だけ投げる。まとめて質問攻めにしない。ユーザーの答えを受けて、次に一番効く問いを選ぶ。
- 質問は「未解決で、かつ実装の方向を分岐させる論点」に絞る。すでに答えが明らかなことや、後から決めても困らない瑣末な点は聞かない。
- ユーザーの答えから合理的に推測できることは推測し、「〇〇という前提で進めますが合ってますか？」と確認に変える。これで質問数を減らせる。
- AskUserQuestion ツールが使える場面（選択肢が固まっている）では積極的に使うと、ユーザーの負担が減る。自由記述で聞いた方が良い論点は普通に文章で聞く。

### 何を解像度高くするか

機能によって重要な論点は変わるが、たいてい以下が曖昧なまま残りやすい:

- **誰のための機能か / 何が嬉しいのか**（ユーザーストーリーの主語と目的）
- **トリガーと入口**（どの画面・どの操作から始まるか）
- **スコープの境界**（今回やること / やらないこと。これが後のサブ Issue 分割を決める）
- **データと永続化**（何を保存するか、既存モデルとの関係）
- **エッジケースと失敗時の挙動**（空・エラー・オフライン・権限なし等）
- **完了の定義**（どうなったら「できた」と言えるか）

### 深掘りをやめるタイミング

「ユーザーストーリーが 1 文で書けて、受け入れ条件が箇条書きで列挙できる」状態になったら、それ以上は聞かない。
完璧主義で質問を続けるより、固まったら次に進む。実装中に出てくる細部は Issue のコメントや後続作業で詰めればよい。

深掘りが一段落したら、固まった理解を**短く要約して提示**し、「この理解で Issue を作ります」と一言添えてからフェーズ 2 へ進む。（ユーザーは即作成を望んでいるので、長い承認待ちはしない。ただし大きな認識違いを早期に拾うため、要約の提示だけは行う。）

## フェーズ 2: 親 Issue を組み立てる

ユーザーストーリー＋受け入れ条件の形でまとめる。本文は**日本語**で書く。

### テンプレート

```markdown
## ユーザーストーリー

〇〇（ユーザー種別）として、
△△したい。
なぜなら □□ だから。

## 背景・目的

（なぜこの機能が必要か。1〜3 文）

## 受け入れ条件

- [ ] 条件1（観測可能・検証可能な形で）
- [ ] 条件2
- [ ] 条件3

## スコープ外

- （今回はやらないこと。境界を明示しておくと後で揉めない）

## 実装サブタスク

（フェーズ 3 で決定。作成後に sub-issue へのリンクで埋まる）
```

受け入れ条件は「実装したかどうかを後から客観的に判定できる」粒度で書く。「使いやすくする」のような主観的表現は避け、「〇〇ボタンを押すと △△ が表示される」のように観測可能な形にする。

## フェーズ 3: サブ Issue に分割する

親 Issue を、**それぞれが独立した 1 つの PR として完結・レビューできる**実装単位に割る。

### 「大きすぎる」の判断基準（切り出すサイン）

以下に当てはまる塊は、さらに小さく切り出す:

- 複数のレイヤーをまたぐ（例: データモデル＋API＋UI が 1 つに混ざっている → レイヤーごとに分ける）
- 独立して出荷できる受け入れ条件が複数含まれている
- レビュアーが一度に読むには差分が大きくなりそう（目安: ざっくり半日〜1 日で終わらない、PR が肥大化しそう）
- 「これとこれは別々にテストできる」と感じる

### 良い分割の指針

- 各サブ Issue は**それ単体で意味のある進捗**になるようにする（「設定ファイルを 1 行足すだけ」のような細かすぎる分割は逆に管理コストが勝つ）。
- 依存順に並べる（土台 → その上に乗るもの）。依存がある場合は本文に「#親番号 / 先行: #xx」と書いておくと進めやすい。
- 各サブ Issue にも軽い受け入れ条件を付ける。親より粒度は粗くてよい。

### サブ Issue テンプレート

```markdown
## 概要

（このサブタスクで何を実装するか。1〜2 文）

## やること

- 具体的な実装項目1
- 具体的な実装項目2

## 受け入れ条件

- [ ] 条件1
- [ ] 条件2

親 Issue: #（親の番号）
```

分割案ができたら、**親 + サブ Issue のタイトル一覧を提示**してから作成に移る。これで全体像をユーザーが一目で確認できる。

## フェーズ 4: GitHub へ作成・リンク

ここは手順が決まっているので、`references/github-issues.md` の手順に従って淡々と実行する。fiddly な sub-issue のリンクは同梱スクリプトに任せる。スクリプトは**プラグインルートからの絶対パス** `${CLAUDE_PLUGIN_ROOT}/skills/feature-planning/scripts/link_sub_issue.sh` で呼ぶ（`scripts/...` の相対パスは作業ディレクトリから解決できず失敗する。`${CLAUDE_PLUGIN_ROOT}` はプラグイン実行時にこのプラグインのルートへ自動展開される）。

大まかな流れ:

1. **ラベルを取得** — `gh label list` で既存ラベルを見て、機能の性質に合うものを選ぶ（例: `enhancement`、プラットフォーム系ラベルなど）。**新しいラベルは勝手に作らない**。既存に適切なものがなければラベルなしでよい。
2. **親 Issue を作成** — `gh issue create` でストーリー本文・ラベルを付けて作成。**返り値の URL 末尾から Issue 番号を取る**（`gh issue list` で直近を取りに行くと競合で取り違える）。
3. **各サブ Issue を作成** — 同様に作成し、各返り値 URL から番号を控える。
4. **sub-issue としてリンク** — 各サブ Issue を `bash ${CLAUDE_PLUGIN_ROOT}/skills/feature-planning/scripts/link_sub_issue.sh <親番号> <子番号>` で親に紐づける（GitHub ネイティブの sub-issue 機能）。
5. **親本文を更新**（任意）— 「実装サブタスク」セクションにサブ Issue へのチェックリストを追記しておくと、ネイティブ sub-issue 表示と二重で見やすい。

詳細なコマンドとエラー対処は `references/github-issues.md` を参照。

## フェーズ 5: 報告

作成し終えたら、簡潔にまとめて報告する:

- 親 Issue へのリンク（番号 + タイトル）
- サブ Issue のリンク一覧
- 付与したラベル
- もし sub-issue リンクに失敗したものがあれば、その旨と代替手段（本文チェックリストで対応済み等）

長々と振り返らない。リンクが並んでいて次に何から着手すべきかが分かれば十分。

## 設計上の注意（移植性）

このスキルを別プロジェクトにコピーして使えるよう、以下を守る:

- リポジトリ名・オーナー・ラベル・マイルストーンは**実行時に `gh` から取得**し、ハードコードしない。
- 特定の言語・フレームワーク（Flutter 等）に依存した前提を本文に埋め込まない。文脈はステアリングや README から都度読む。
- ステアリングや Kiro 関連ファイルは「あれば読む」程度の任意参照に留め、無くても全フェーズが成立するようにする。
